# 简历与面试指南

## 1. 简历项目描述

**KnowledgeFlow AI 企业级智能知识库问答系统**  
技术栈：Python、FastAPI、Pydantic、PyMuPDF、Sentence Transformers、ChromaDB、Ollama/Qwen、Streamlit、pytest、Docker

- 从零设计并实现完整 RAG Pipeline，支持 PDF/TXT/Markdown 解析、文本清洗与重叠切分、批量 Embedding、ChromaDB 持久化检索和本地 Qwen 生成。
- 设计带文档名、页码、Chunk ID、相似度和原文片段的可追溯回答，并通过最低相关阈值和上下文约束实现无答案拒答，降低模型幻觉。
- 使用 FastAPI/Pydantic 提供文档上传、更新、删除、知识库构建和问答接口，增加文件大小/类型校验、统一异常、日志轮转和配置管理。
- 构建 10 条带标签的 RAG 评估集；在演示集上实测 Recall@4、Top-1 来源准确率和无答案拒答准确率均为 100%，平均检索延迟 24.84 ms，并明确记录小样本局限。
- 编写 20 项自动化测试覆盖文档加载、文本切分、Embedding、向量检索、RAG 拒答和 API 生命周期，提供 Streamlit 界面及 Docker 配置。

> 面试时必须主动补充：100% 是小型自建演示集上的结果，不代表生产环境。诚实解释指标边界比单纯展示高分更加分。

## 2. 项目亮点

1. **没有过早依赖 LangChain**：亲自实现每个 RAG 环节，能解释数据怎样从文档流到答案。
2. **检索和生成分离**：先用命令行验证召回，再接大模型，定位问题时能判断是“没找到”还是“没答好”。
3. **来源可追溯**：页码和 Chunk 元数据贯穿解析、存储、检索和最终响应。
4. **拒答机制**：相似度不足时直接返回找不到依据，不让模型凭常识补全。
5. **可评估而非凭感觉**：提供数据集、指标、基线、延迟和限制声明。
6. **工程闭环**：配置、日志、异常、API、UI、测试、Git 历史和 Docker 配置完整。

## 3. 主要技术难点及回答思路

### 难点一：如何确定 Chunk 大小

Chunk 太小会丢失上下文，太大会混入无关内容、降低检索精度并占用更多上下文。本项目默认 500 字符、重叠 80 字符，先保证跨边界语义连续，再通过评估集调参。更严格的生产系统应按 Token、标题结构和文档类型分别评估。

### 难点二：如何减少无依据回答

采用两道门：第一道是检索最低分数，低于阈值直接拒答；第二道是 Prompt，要求模型只能引用编号上下文。最终响应还返回来源，便于用户核验。必须强调 Prompt 不能提供绝对安全保证。

### 难点三：文档更新后怎样防止旧答案残留

更新流程先完整校验并保存新文件，再删除旧文件和旧文档 ID 对应的向量。这样无效新文件不会破坏原数据，成功更新后旧 Chunk 又不会继续参与检索。

### 难点四：Windows 与依赖兼容

项目先实际验证 Python 3.14 下 PyTorch、Sentence Transformers、ChromaDB、PyMuPDF、FastAPI 和 Streamlit，再固定直接依赖版本。Docker 是可选部署方式，Ollama 继续运行在宿主机，避免把数 GB 模型打进 API 镜像。

## 4. 两分钟项目介绍

“我做的是 KnowledgeFlow AI，一个面向企业文档的本地 RAG 知识库系统。它解决的问题是：通用大模型不知道公司内部资料，而且容易在没有依据时编造。

系统支持上传 PDF、TXT 和 Markdown。我先用 PyMuPDF 或普通文本读取保留页码和文档 ID，再手写清洗与重叠切分，把每个 Chunk 用 multilingual-e5-small 转成 384 维向量并持久化到 ChromaDB。提问时，系统把问题向量化，执行 Top-K 语义检索，再用最低分数过滤不相关内容。只有检索到可靠依据时，才将带编号来源的上下文交给本地 Ollama Qwen3 4B，最后返回回答、文档名、页码、相似度和原文片段。

工程上我用 FastAPI 和 Pydantic 实现文档生命周期及聊天 API，用 Streamlit 做演示界面，并加入集中配置、轮转日志、统一异常、文件安全校验和 Docker 配置。我还构造了 10 条标签数据做基础评估，并写了 20 项自动化测试。这个项目最重要的收获是，我能清楚区分检索问题与生成问题，也知道高指标必须结合数据集规模和业务分布解释。”

## 5. 高频追问与参考回答

### Q1：什么是 RAG？为什么不直接问大模型？

RAG 是“检索增强生成”。先从外部知识库找依据，再让模型基于依据组织语言。它不用重新训练模型就能更新知识，能提供来源，也能减少幻觉；但效果依赖文档质量和检索质量。

### Q2：Embedding 和大语言模型有什么区别？

Embedding 模型的输出主要是固定长度向量，用于比较文本相似性；大语言模型输出 Token 序列，用于生成文字。这个项目使用两个不同模型分别完成“找资料”和“写答案”。

### Q3：为什么不能把整本 PDF 放进 Prompt？

上下文窗口有限，全文会增加延迟和推理成本，还会稀释关键信息。先检索少量相关 Chunk，可以让模型更集中地使用证据。

### Q4：Chunk overlap 有什么用？

一个观点可能正好跨越切分边界。相邻 Chunk 保留一小段重叠，可降低语义被截断的概率；但重叠太大会增加重复向量和重复上下文。

### Q5：Top-K 和最低阈值有什么区别？

Top-K 永远选择排名最高的 K 个，即使所有结果都不相关。最低阈值判断结果是否“足够相关”。二者组合后，既限制上下文数量，又能对知识库之外的问题拒答。

### Q6：什么是余弦相似度？

它比较两个向量方向是否接近，不太受向量长度影响。值越大通常表示语义越相似。归一化向量后，点积与余弦相似度等价。

### Q7：ChromaDB 为什么不能用 MySQL 代替？

MySQL 擅长结构化字段、事务和精确条件查询；ChromaDB 专门保存高维向量并进行近邻搜索。向量数据库不是替代所有传统数据库，两者在生产系统中经常同时存在。

### Q8：ChromaDB 和 FAISS 有什么区别？

FAISS 更像高性能向量索引库，需要自己管理文本、元数据和持久化流程；ChromaDB 提供集合、文档、元数据、过滤和持久化等更完整的数据库体验。本地作品项目使用 ChromaDB 更容易形成闭环。

### Q9：为什么使用 FastAPI？

它支持类型注解和 Pydantic 校验，能自动生成 OpenAPI/Swagger 文档，异步接口写法清晰，适合 Python AI 服务。框架本身不会让模型更聪明，它解决的是服务化和数据契约问题。

### Q10：如何评价 RAG？

要拆成检索与生成。检索可看 Recall@K、Top-1 命中和延迟；生成可看 Faithfulness、Answer Relevance、引用正确性和拒答能力。不能只挑几道题凭主观感觉判断。

### Q11：Recall@4 等于 1 说明什么？

在当前 7 个可回答问题中，正确来源都出现在前 4 个结果里。它不代表答案一定正确，也不代表换一批业务问题仍是 1，所以必须同时说明评估集规模和分布。

### Q12：单元测试怎样避免真的调用大模型？

给 Pipeline 注入可控的假检索器和假 Ollama 客户端，验证输入输出与分支逻辑。真实模型用于少量集成或手工验收，避免测试慢、不稳定和依赖外部服务。

### Q13：如何处理扫描 PDF？

当前 PyMuPDF 只能直接提取文本层。扫描件需要先检测空文本页，再调用 OCR，例如 PaddleOCR 或 Tesseract，并为 OCR 误差增加清洗与质量标记。这是明确的后续扩展点。

### Q14：Prompt Injection 怎么防？

把文档内容视为不可信数据，在系统指令中明确忽略文档内命令；限制工具和数据权限；做输入输出过滤与审计。仅靠 Prompt 无法彻底防护，生产系统需要多层安全控制。

### Q15：如果检索效果差，你会先查什么？

先用 `scripts/search.py` 看原始召回：检查解析文本、Chunk 边界、Embedding 前缀、向量归一化、Top-K 和阈值。只有召回正确但回答不好时，才重点调整 Prompt 或生成模型。

## 6. 诚实表达边界

- 不说“企业级”就等于已经支持真实企业的权限、审计和高可用；这里指工程结构和企业文档场景。
- 不把小数据集的 100% 描述为行业领先。
- 不声称 Docker 构建成功，除非实际安装 Docker 并执行构建。
- 不声称 RAG 消除了幻觉，应说“降低并提供核验依据”。
- 能明确指出限制和验证方案，通常比堆叠框架名更能证明工程判断力。
